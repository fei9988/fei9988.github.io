# [Gson] JSON 反序列化兼容性问题修复笔记

> 当你维护一个遗留系统并进行数据结构升级（Schema Evolution）时，经常会遇到新旧数据格式不一致、日期格式不标准等问题。本笔记总结了如何优雅地解决这些报错。

---


> MySQL 中存储的 json 字符串，对象中有字段为字符串，实际是对象，现在需要改为对象，此时数据库中已经保存了很多数据（带有转移字符），直接修改为对象接收会报错，如何处理？

- 举例
  - 原本有类 A，其中有字段 people 为 String 类型
  ```java
    public class A{
      private String people;
    }
  ```
  - 数据库中 people 字段存储的 json 字符串为 `{"name":"张三","age":18,"address":\"{\"province\":\"北京\",\"city\":\"北京\"}}`
  - 现在想把类 A 修改为
  ```java
    public class A{
      private People people;
    }
  
    public class People {
      private String name;
      private int age;
      private Address address;
    }
  
    public class Address {
      private String province;
      private String city;
    }
  ```
  - 直接序列化会报错，因为数据库中存储的 json 字符串中，address 字段是一个字符串，而不是一个对象


# 解决方法

## 1. 问题背景
在 Spring Boot 项目中对旧有数据结构进行重构。
* **变更前**：某字段（如 `people`）是 JSON 字符串格式（`String`），存储内容是转义后的 JSON。
* **变更后**：将该字段改为强类型的 Java 对象（`Object`），期望 JSON 结构是嵌套对象。
* **冲突点**：数据库中存量的旧数据仍是 String，新数据是 Object。直接反序列化会报错。
* **附加问题**：日期字段使用了无时区的 ISO 格式（`yyyy-MM-dd'T'HH:mm:ss`），导致默认 Gson 解析失败。

---

## 2. 核心问题一：String 与 Object 的兼容 (Schema Evolution)

**报错现象：**
`Expected BEGIN_OBJECT but was STRING`

**解决方案：**
使用自定义 `TypeAdapter` (`JsonDeserializer`) 来接管解析逻辑。判断 JSON 元素是基本类型（String）还是对象（Object），做分流处理。

### 核心代码实现

```java
import com.google.gson.*;
import java.lang.reflect.Type;

public class CompatibleTypeAdapter implements JsonDeserializer<MyTargetObject> {

    @Override
    public MyTargetObject deserialize(JsonElement json, Type typeOfT, JsonDeserializationContext context) 
            throws JsonParseException {
        
        // 【情况 A】兼容旧数据：字段是字符串
        // 例如："{\"key\":\"value\"}"
        if (json.isJsonPrimitive() && json.getAsJsonPrimitive().isString()) {
            String jsonString = json.getAsString();
            // 关键：将提取出的字符串，再次作为 JSON 解析
            return new Gson().fromJson(jsonString, MyTargetObject.class);
        }
        
        // 【情况 B】标准新数据：字段是对象
        // 例如：{"key":"value"}
        if (json.isJsonObject()) {
            // 使用默认 Gson 逻辑解析标准对象
            return new Gson().fromJson(json, MyTargetObject.class);
        }

        return null;
    }
}
```

## 3. 核心问题二：日期格式无时区报错

**报错现象：**
`java.text.ParseException: ... ["2026-01-22T16:35:08"]: No time zone indicator`

**原因分析：**
Gson 处理 `java.util.Date` 时，默认期望标准的 ISO 8601 格式（包含 `Z` 或 `+08:00` 时区标识）。如果时间字符串类似 `2026-01-22T16:35:08`（缺时区），就会抛出此异常。

**解决方案：**
在构建 Gson 时显式指定日期格式，忽略时区检查。

```java
// 在 GsonBuilder 中设置
.setDateFormat("yyyy-MM-dd'T'HH:mm:ss")
```

> **注意**：如果是 Java 8 的 `LocalDateTime` 类型，则需要注册专门的 Adapter，因为 Gson 默认不支持 Java 8 时间 API。

---

## 4. 综合配置 (Best Practice)

将上述两个问题的解决方案整合到一个 `Gson` 实例中：

```java
public Gson getCompatibleGson() {
    return new GsonBuilder()
        // 1. 解决 String 变 Object 的结构兼容问题
        .registerTypeAdapter(BusinessInfo.class, new CompatibleTypeAdapter())
        
        // 2. 解决日期缺少时区的问题
        .setDateFormat("yyyy-MM-dd'T'HH:mm:ss")
        
        // 3. (可选) 序列化空字段，防止 null 字段消失
        // .serializeNulls() 
        
        .create();
}
```

**使用方式：**
```java
Gson gson = getCompatibleGson();
MyData data = gson.fromJson(jsonString, MyData.class);
```

## 5. 延伸思考：Jackson 方案

如果是 Spring Boot 项目，推荐优先使用 **Jackson**（Spring 默认组件），处理兼容性更简单：

* **String/Object 兼容**：使用 `@JsonCreator` 注解构造函数即可，无需写复杂的 Adapter。
* **日期格式**：使用 `@JsonFormat(pattern = "...")` 注解即可。

```java
public class MyObject {
    
    // 兼容 String 输入的构造函数
    @JsonCreator
    public MyObject(String str) throws IOException {
        // 手动解析或由 Jackson 自动转换
    }
}
```
