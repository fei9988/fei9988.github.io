# SSE 流式协议与心跳设计

## 这个设计解决什么问题

很多流式系统一开始只是“把文本一点点往前端推”，但上线后很快会遇到几个真实问题：

- 正文、进度、错误混在一起，前端不好渲染
- Markdown、空格、换行在 SSE 里被破坏
- 响应开始后再报错，HTTP 状态码已经没法表达失败
- 客户端断开连接后，服务端很久才感知到取消

一个成熟的 SSE 设计，重点不是“能流”，而是定义一套稳定、可扩展、可观测的事件协议。

## 亮点一：把不同语义拆成不同事件

推荐把流式输出拆成几类事件，而不是所有东西都塞进 `message`：

- `message`：最终答案的正文增量
- `progress`：过程态提示，例如“正在检索”“正在执行工具”
- `error`：流内错误
- `done`：正常结束信号
- `keep_alive`：心跳

这样做的价值很直接：

- 前端能把正文和过程态分开渲染
- 错误处理逻辑统一
- 协议层更容易扩展

## 亮点二：用 JSON 包裹文本，保护格式

SSE 的 `data:` 对纯文本并不友好，尤其是 Markdown 和多行内容。一个实用做法是不要直接发送裸文本，而是统一包装成 JSON：

```json
{"text":"### 标题\n- 列表项\n"}
```

这样有几个好处：

- 前导空格和换行由 JSON 转义保护
- 前端统一 `JSON.parse`，逻辑简单
- 后续要给文本加元数据也容易扩展

一个简化示例如下：

```java
ServerSentEvent<String> message(String chunk) {
    return ServerSentEvent.<String>builder()
            .event("message")
            .data("{\"text\":\"" + escape(chunk) + "\"}")
            .build();
}
```

如果你的前端会渲染 Markdown，这个设计通常比直接发送裸字符串稳定很多。

## 亮点三：把错误也纳入流协议

SSE 一旦开始写响应，后面就不能再可靠地表达 HTTP 失败状态了。所以更稳的做法是：把运行期错误转成一条 `error` 事件发给前端。

示例：

```java
Flux<ServerSentEvent<String>> stream = businessFlux
        .map(this::toMessageEvent)
        .onErrorResume(ex -> Flux.just(errorEvent(ex)));
```

这种设计的优点是：

- 前端只需要处理“事件”，不需要区分错误发生在响应前还是响应后
- 服务端可以在任意阶段兜底
- 成功路径和失败路径都有统一协议

## 亮点四：用具名心跳提升断连感知

很多框架在客户端断开连接后，不会立刻取消上游流，而是等到下一次写数据失败才感知到断开。对于模型推理、工具调用这类中间有空档的流程，这会带来明显延迟。

比较稳的方案是给正文流叠加一个低频心跳，例如每 5 秒发一条：

```text
event: keep_alive
data: {}
```

它的作用不是业务展示，而是：

- 定期制造写操作
- 更快发现连接已关闭
- 减少“前端早就走了，后端还在跑”的浪费

建议心跳也用具名事件，而不是裸注释帧。这样前端按未知事件忽略即可，不容易和正文解析逻辑打架。

## 亮点五：把 `done` 当成协议级终止信号

一个好用的经验是：正常完成时显式发送 `done`，异常时只发 `error`，不要两者都发。

这样前端状态机非常清楚：

- 收到 `message` / `progress`：继续处理中
- 收到 `done`：本轮成功结束
- 收到 `error`：本轮失败结束

这类清晰的协议约束，往往比复杂的前端兼容逻辑更有长期价值。

## 为什么这是通用亮点

这套设计的可复用点在于，它把流式响应从“文本搬运”提升成了“事件协议设计”：

- 有明确的事件边界
- 能保护富文本格式
- 能承载运行期错误
- 能更快感知断连
- 能支持前后端独立演进

## 适用场景

- LLM 对话
- 流式搜索 / 检索结果展示
- 长耗时工作流的前端进度反馈
- 需要同时返回正文和过程态的在线服务

## 使用建议

1. 不要把所有输出都塞进 `message`
2. 文本尽量用 JSON 包装
3. 错误用流内事件表达
4. 心跳频率不要太高，通常 3 到 10 秒够用
5. `done` 和 `error` 保持互斥
