# LeafChatModelLoggingObservationHandler 日志机制分享

## 文档目的

`LeafChatModelLoggingObservationHandler` 这套日志机制的设计目的、核心作用、运行方式，以及它和 `SimpleLoggerAdvisor` 的区别。

希望回答四个问题：

- 为什么要单独做这套日志
- 它解决了什么问题
- 它是怎么避免日志重复的
- 实际排查问题时应该怎么看这类日志

## 一句话结论

`LeafChatModelLoggingObservationHandler` 的核心策略是：

> 每次真实模型调用都打印 prompt，但只让叶子节点打印 completion。

这样做的好处是：

- 能完整看到 Spring AI 底层模型调用链
- 能看清 tool call 和 tool response 是否进入 prompt
- 能避免最终答案在递归调用场景下重复打印
- 更适合分析模型为什么得到当前答案

## 背景问题

在 Spring AI 的 tool calling 场景里，一次表面上的 `ChatClient` 调用，底层往往不止一次模型请求。

典型流程如下：

1. 模型第一次收到用户问题
2. 模型判断需要调用工具
3. 框架执行工具
4. 模型第二次基于工具结果生成最终答案

也就是说，业务上看起来是一次请求，底层可能是这样：

```text
ChatClient 一次调用
  -> ChatModel 调用 A：判断是否调工具
  -> Tool 执行
  -> ChatModel 调用 B：基于工具结果生成答案
```

如果每一层调用都打印 completion，就会出现：

- 最终答案重复打印
- 日志噪音大
- 很难判断哪一次模型调用才是真正产出最终回答的那次

`LeafChatModelLoggingObservationHandler` 就是为了解决这个问题。

## 类的定位

这个类是一个 Micrometer Observation Handler，工作在 `ChatModelObservationContext` 层，而不是 `ChatClient` 包装层。

源码位置：

- `src/main/java/com/bytedance/ai/config/LeafChatModelLoggingObservationHandler.java`
- `src/main/java/com/bytedance/ai/config/OpenAiChatModelLoggingConfig.java`

类定义如下：

```java
public class LeafChatModelLoggingObservationHandler
        implements ObservationHandler<ChatModelObservationContext> {
```

它的意思是：

- 只处理 chat model 相关的 observation
- 能拿到每一次真实模型调用的 request 和 response
- 能参与每次模型调用的开始和结束阶段

## 注册方式

该类通过 Spring Bean 注册：

```java
@Configuration
public class OpenAiChatModelLoggingConfig {

    @Bean
    public ObservationHandler<ChatModelObservationContext> leafChatModelLoggingObservationHandler() {
        return new LeafChatModelLoggingObservationHandler();
    }
}
```

这意味着只要底层发生 `ChatModelObservationContext`，这个 handler 就会自动参与生命周期。

## 为什么不只靠 SimpleLoggerAdvisor

项目里已经有 `SimpleLoggerAdvisor`，例如：

```java
.defaultAdvisors(
        new SimpleLoggerAdvisor(),
        MessageChatMemoryAdvisor.builder(conversationChatMemory).build())
```

它的特点是：

- 工作在 `ChatClient` 层
- 更适合看一次业务级调用的整体输入输出

但在 tool calling 场景下，它不够细：

- 看不清底层到底调了几次模型
- 看不清 tool call 和 tool response 是如何进入 prompt 的
- 可能只能看到整体输入和最终输出，看不到中间过程

`LeafChatModelLoggingObservationHandler` 则补足了这部分能力，它关注的是：

- 每一次真实模型调用
- 每一次模型调用的 prompt
- 哪一次调用产出了最终 completion

## 核心思路

这套日志机制只有两条规则：

1. `onStart()` 时打印当前模型调用的 prompt
2. `onStop()` 时只让叶子节点打印 completion

关键词是“叶子节点”。

叶子节点指的是：

- 当前 observation 没有再触发更深层的子 observation
- 它就是这棵模型调用树最末端的那次调用

所以最终效果是：

- 外层调用：打印 prompt，不打印 completion
- 内层叶子调用：打印 prompt，也打印 completion

## 核心代码解析

### 1. 类定义与标记位

```java
public class LeafChatModelLoggingObservationHandler implements ObservationHandler<ChatModelObservationContext> {

    private static final Logger log = LoggerFactory.getLogger(LeafChatModelLoggingObservationHandler.class);

    private static final String HAS_CHILD_OBSERVATION = LeafChatModelLoggingObservationHandler.class.getName()
            + ".hasChildObservation";
```

这里定义了一个上下文标记 `HAS_CHILD_OBSERVATION`。

用途是：

- 如果某个 observation 有子调用
- 就给它打一个标记
- 这样它结束时就知道自己不是叶子节点

### 2. `onStart()`：打印 prompt，并给父节点打标

```java
@Override
public void onStart(ChatModelObservationContext context) {
    markParentHasChild(context);
    log.info("Chat Model Prompt Content:\n{}", String.join("\n", prompt(context)));
}
```

这里做了两件事：

- 调 `markParentHasChild(context)`，通知父 observation “你有孩子”
- 打印当前这次模型调用的 prompt

这保证了每一次真实模型调用的输入都能在日志里看到。

### 3. `onStop()`：只打印叶子节点 completion

```java
@Override
public void onStop(ChatModelObservationContext context) {
    if (Boolean.TRUE.equals(context.get(HAS_CHILD_OBSERVATION))) {
        return;
    }
    List<String> completion = completion(context);
    if (!CollectionUtils.isEmpty(completion)) {
        log.info("Chat Model Completion:\n{}", ObservabilityHelper.concatenateStrings(completion));
    }
}
```

这段逻辑是整个类的核心：

- 如果当前节点被标记为“有子节点”，说明它不是叶子节点
- 那么直接返回，不打印 completion
- 只有没有子节点的调用，才打印最终输出

这就避免了同一个最终答案在外层和内层 observation 中被重复打印。

### 4. `markParentHasChild()`：子调用反向标记父调用

```java
private void markParentHasChild(ChatModelObservationContext context) {
    ObservationView parentObservation = context.getParentObservation();
    if (parentObservation == null) {
        return;
    }
    if (parentObservation.getContextView() instanceof Observation.Context parentContext) {
        parentContext.put(HAS_CHILD_OBSERVATION, true);
    }
}
```

这是整个机制最关键的设计点。

它的思路很简单：

- 当前 observation 开始时，先找父 observation
- 如果找到了，说明当前调用是嵌套调用
- 那就给父 observation 打标：`HAS_CHILD_OBSERVATION = true`

这样父节点在 `onStop()` 时就能知道：

- 自己下面还有子调用
- 不应该重复打印 completion

### 5. `supportsContext()`：只处理 ChatModel observation

```java
@Override
public boolean supportsContext(Observation.Context context) {
    return context instanceof ChatModelObservationContext;
}
```

这个方法保证 handler 只处理 chat model 相关 observation，不会误处理其他类型事件。

## Prompt 日志是怎么来的

prompt 内容来自当前模型请求的 `instructions`：

```java
private List<String> prompt(ChatModelObservationContext context) {
    if (CollectionUtils.isEmpty(context.getRequest().getInstructions())) {
        return List.of();
    }
    return context.getRequest().getInstructions().stream().map(this::formatMessage).toList();
}
```

这里拿到的不是业务代码里表面上的一行 `.user(message)`，而是底层真正发给模型的完整消息列表。

所以在 tool calling 场景下，它能展示：

- system prompt
- user message
- assistant tool call
- tool response

这比普通业务日志更贴近“模型真实看到的内容”。

## 为什么要自定义格式化消息

日志如果只是一股脑把 message 打出来，可读性会很差。所以这里针对不同消息类型做了专门格式化。

### `AssistantMessage` + tool call

```java
if (message instanceof AssistantMessage assistantMessage && assistantMessage.hasToolCalls()) {
    return assistantMessage.getToolCalls()
            .stream()
            .map(toolCall -> "ASSISTANT_TOOL_CALL: name=%s, id=%s, arguments=%s"
                    .formatted(toolCall.name(), toolCall.id(), toolCall.arguments()))
            .reduce((left, right) -> left + "\n" + right)
            .orElse("ASSISTANT: " + message.getText());
}
```

日志效果类似：

```text
ASSISTANT_TOOL_CALL: name=KnowledgeTool, id=call_123, arguments={"query":"A 产品 开通 权限"}
```

这样可以快速看出：

- 模型调了哪个工具
- 工具调用 id 是什么
- 工具入参是什么

### `ToolResponseMessage`

```java
if (message instanceof ToolResponseMessage toolResponseMessage) {
    return toolResponseMessage.getResponses()
            .stream()
            .map(response -> "TOOL_RESPONSE: name=%s, id=%s, responseData=%s"
                    .formatted(response.name(), response.id(), response.responseData()))
            .reduce((left, right) -> left + "\n" + right)
            .orElse("TOOL_RESPONSE: <empty>");
}
```

日志效果类似：

```text
TOOL_RESPONSE: name=KnowledgeTool, id=call_123, responseData=...
```

这样可以确认：

- 工具是否真的执行了
- 工具返回了什么
- 工具返回值有没有进入模型后续推理

### 普通消息

```java
return "%s: %s".formatted(message.getMessageType().name(), message.getText());
```

普通消息会打成：

```text
SYSTEM: ...
USER: ...
ASSISTANT: ...
```

整体阅读体验会比较自然。

## Completion 日志是怎么来的

completion 取自模型 response：

```java
private List<String> completion(ChatModelObservationContext context) {
    if (context.getResponse() == null || CollectionUtils.isEmpty(context.getResponse().getResults())) {
        return List.of();
    }
    return context.getResponse()
            .getResults()
            .stream()
            .filter(generation -> generation.getOutput() != null
                    && StringUtils.hasText(generation.getOutput().getText()))
            .map(generation -> generation.getOutput().getText())
            .toList();
}
```

最后在叶子节点的 `onStop()` 中打印：

```text
Chat Model Completion:
最终回答内容...
```

这里有两个重点：

- 只打印有文本内容的 generation
- 只让叶子节点打印，避免重复

## 一个典型例子：QA + KnowledgeTool

假设用户问题是：

```text
A 产品怎么开通权限？
```

在 `QaCapability` 场景下，底层可能发生这样的调用树：

```text
Observation A
  - 模型收到 system + history + user
  - 模型决定调用 KnowledgeTool

Observation B
  - 模型收到 system + history + user + tool call + tool response
  - 模型生成最终答案
```

### 第一次模型调用日志

```text
Chat Model Prompt Content:
SYSTEM: 你是一个知识库问答助手...
USER: A 产品怎么开通权限？
```

这表示模型第一次拿到的输入。

### 第二次模型调用日志

```text
Chat Model Prompt Content:
SYSTEM: 你是一个知识库问答助手...
USER: A 产品怎么开通权限？
ASSISTANT_TOOL_CALL: name=KnowledgeTool, id=call_123, arguments={"query":"A 产品 开通 权限"}
TOOL_RESPONSE: name=KnowledgeTool, id=call_123, responseData=【知识库检索结果】...
```

这说明：

- 模型确实决定调工具
- 工具也确实返回了结果
- 返回结果已经被喂回了第二次模型调用

### 最终 completion

```text
Chat Model Completion:
A 产品开通权限需要进入 xx 平台，申请对应角色，并联系管理员审批。
```

因为只有第二次调用是叶子节点，所以最终答案只会打印一次。

## 这套日志的实际价值

### 1. 看清模型调用链

它能帮助我们确认：

- 底层到底调了几次模型
- 哪一次调用触发了工具
- 哪一次调用产出了最终回答

### 2. 看清 prompt 演进过程

它能直接展示：

- system prompt 是否生效
- user 输入是否正确
- tool call 是否进入上下文
- tool response 是否进入上下文

### 3. 看清工具调用是否正常

它特别适合排查：

- 模型为什么没有调工具
- 模型调错了工具
- 工具入参不对
- 工具返回空数据
- 工具返回了数据但没有参与最终回答生成

### 4. 降低日志噪音

通过“只打印叶子 completion”，它避免了：

- 同一份答案重复刷屏
- 外层 observation 和内层 observation 重复输出
- 日志可读性下降

## 和 SimpleLoggerAdvisor 的区别

| 对比项 | SimpleLoggerAdvisor | LeafChatModelLoggingObservationHandler |
| --- | --- | --- |
| 所在层级 | ChatClient 层 | ChatModel Observation 层 |
| 关注对象 | 一次业务级 ChatClient 调用 | 每一次真实模型调用 |
| 工具链中间过程可见性 | 一般 | 很强 |
| 是否容易看到 tool call / tool response | 不完整 | 完整 |
| completion 打印策略 | 以整体调用为主 | 仅叶子节点打印 |

可以记成：

- `SimpleLoggerAdvisor`：看业务请求整体
- `LeafChatModelLoggingObservationHandler`：看模型执行细节

两者是互补关系，不是替代关系。

## 分享时建议强调的点

如果这份文档用于组内分享，建议重点强调下面几点：

- 这不是普通业务日志，而是模型调用链路日志
- 它解决的是 tool calling 场景下“看不清过程”和“completion 重复”的问题
- 它的关键设计点是“子调用给父调用打标”
- 它最大的价值是看 prompt 如何一步步演进
- 它非常适合定位工具调用、prompt 拼装、最终答案来源问题

## 口头分享版总结

可以直接复述下面这段：

> 我们增加 `LeafChatModelLoggingObservationHandler`，不是为了再多打一份日志，而是为了把 Spring AI 底层真实模型调用过程看清楚。  
> 在 tool calling 场景下，一次 ChatClient 调用底层可能会递归触发多次模型调用。  
> 如果每层都打 completion，最终答案会重复很多次。  
> 所以我们采用的策略是：每次模型调用都打印 prompt，但只让叶子节点打印 completion。  
> 这样既能完整看到工具调用前后 prompt 的变化，又能避免最终答案重复刷屏。  
> 对排查“模型为什么调工具”“工具结果有没有喂回去”“最终回答怎么生成的”非常有帮助。

## 最终总结

`LeafChatModelLoggingObservationHandler` 的核心价值可以概括为四点：

- 提升底层模型调用可观测性
- 清晰展示 tool calling 链路
- 避免 completion 重复打印
- 为大模型问题排查提供高价值日志依据

一句话总结：

> 它让我们看到“模型实际上是怎么一步一步得到最终答案的”。
```java
import java.util.List;

import io.micrometer.observation.Observation;
import io.micrometer.observation.ObservationHandler;
import io.micrometer.observation.ObservationView;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.springframework.ai.chat.messages.AssistantMessage;
import org.springframework.ai.chat.messages.Message;
import org.springframework.ai.chat.messages.ToolResponseMessage;
import org.springframework.ai.chat.observation.ChatModelObservationContext;
import org.springframework.ai.observation.ObservabilityHelper;
import org.springframework.util.CollectionUtils;
import org.springframework.util.StringUtils;

/**
 * Logs every model prompt, but only logs the completion for the leaf model call.
 *
 * <p>In Spring AI's streaming tool-call flow, outer observations aggregate the final
 * response from nested model calls. Logging completion for every observation therefore
 * repeats the same final answer. A child observation marks its parent here, so parent
 * completions can be suppressed while leaf completions remain visible.
 */
public class LeafChatModelLoggingObservationHandler implements ObservationHandler<ChatModelObservationContext> {

    private static final Logger log = LoggerFactory.getLogger(LeafChatModelLoggingObservationHandler.class);

    private static final String HAS_CHILD_OBSERVATION = LeafChatModelLoggingObservationHandler.class.getName()
            + ".hasChildObservation";

    @Override
    public void onStart(ChatModelObservationContext context) {
        markParentHasChild(context);
        log.info("Chat Model Prompt Content:\n{}", String.join("\n", prompt(context)));
    }

    @Override
    public void onStop(ChatModelObservationContext context) {
        if (Boolean.TRUE.equals(context.get(HAS_CHILD_OBSERVATION))) {
            return;
        }
        List<String> completion = completion(context);
        if (!CollectionUtils.isEmpty(completion)) {
            log.info("Chat Model Completion:\n{}", ObservabilityHelper.concatenateStrings(completion));
        }
    }

    @Override
    public boolean supportsContext(Observation.Context context) {
        return context instanceof ChatModelObservationContext;
    }

    private void markParentHasChild(ChatModelObservationContext context) {
        ObservationView parentObservation = context.getParentObservation();
        if (parentObservation == null) {
            return;
        }
        if (parentObservation.getContextView() instanceof Observation.Context parentContext) {
            parentContext.put(HAS_CHILD_OBSERVATION, true);
        }
    }

    private List<String> prompt(ChatModelObservationContext context) {
        if (CollectionUtils.isEmpty(context.getRequest().getInstructions())) {
            return List.of();
        }
        return context.getRequest().getInstructions().stream().map(this::formatMessage).toList();
    }

    private String formatMessage(Message message) {
        if (message instanceof AssistantMessage assistantMessage && assistantMessage.hasToolCalls()) {
            return assistantMessage.getToolCalls()
                    .stream()
                    .map(toolCall -> "ASSISTANT_TOOL_CALL: name=%s, id=%s, arguments=%s"
                            .formatted(toolCall.name(), toolCall.id(), toolCall.arguments()))
                    .reduce((left, right) -> left + "\n" + right)
                    .orElse("ASSISTANT: " + message.getText());
        }
        if (message instanceof ToolResponseMessage toolResponseMessage) {
            return toolResponseMessage.getResponses()
                    .stream()
                    .map(response -> "TOOL_RESPONSE: name=%s, id=%s, responseData=%s"
                            .formatted(response.name(), response.id(), response.responseData()))
                    .reduce((left, right) -> left + "\n" + right)
                    .orElse("TOOL_RESPONSE: <empty>");
        }
        return "%s: %s".formatted(message.getMessageType().name(), message.getText());
    }

    private List<String> completion(ChatModelObservationContext context) {
        if (context.getResponse() == null || CollectionUtils.isEmpty(context.getResponse().getResults())) {
            return List.of();
        }
        return context.getResponse()
                .getResults()
                .stream()
                .filter(generation -> generation.getOutput() != null
                        && StringUtils.hasText(generation.getOutput().getText()))
                .map(generation -> generation.getOutput().getText())
                .toList();
    }
}

```
