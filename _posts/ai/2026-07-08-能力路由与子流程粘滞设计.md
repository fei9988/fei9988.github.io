# 能力路由与子流程粘滞设计

## 这个设计解决什么问题

很多 Agent 系统都有同一个入口，但背后其实包含多种能力：问答、填表、搜索、执行工具、分析、生成结构化结果。问题是，如果所有请求都交给一个大而全的 Prompt，系统会很快变得难以维护。

一个更稳的办法是把系统拆成多个能力单元，再由一层路由器决定“这一轮该走哪个子能力”。

## 亮点一：单入口，内部多能力

这个模式的核心不是“多几个类”，而是把每种能力变成统一抽象，例如：

```java
public interface Capability {
    String name();
    String routingDescription();
    Flux<Event> execute(UserTurn turn);
}
```

这么做的价值是：

- 外部入口稳定
- 内部能力边界清晰
- 新能力可以增量接入

对于 Agent 来说，这比不断在一个巨大 Prompt 和一个巨大工具集里叠逻辑要可控得多。

## 亮点二：能力元信息集中管理

一个常见坑是：路由描述写一份，用户可见介绍写一份，默认能力配置再写一份。时间久了，这些信息就会漂移。

更好的方式是引入一个注册表或目录层，统一维护：

- 能力名称
- 路由描述
- 面向用户的功能介绍
- 默认兜底能力

这样路由器、能力介绍页、监控埋点都能复用同一份元数据，避免重复维护。

## 亮点三：路由失败要能优雅降级

如果你的路由依赖模型分类，那就必须接受一个现实：模型会输出脏结果，甚至偶发失败。

所以一个成熟的路由层应该具备明确的降级链路，例如：

1. 只有一个能力时直接命中
2. 多能力时走分类
3. 分类结果非法时回退到上一次命中的能力
4. 再失败时回退到默认能力

示例伪代码：

```java
String route(String input, String lastCapability) {
    String fallback = lastCapability != null ? lastCapability : "qa";
    try {
        String picked = classify(input);
        return registry.contains(picked) ? picked : fallback;
    } catch (Exception e) {
        return fallback;
    }
}
```

这个设计的重点是：路由层负责稳定性，不把模型输出原样暴露给主流程。

## 亮点四：引入“子流程粘滞”

纯粹每轮独立分类，用户体验通常不够自然。因为用户很多时候并不是“每一轮都换任务”，而是在连续完成同一个子流程。

比如：

- 第一句：帮我生成一个筛选条件
- 第二句：把时间范围改成最近 30 天
- 第三句：再排除测试账号

这三句如果都重新分类，很可能跳来跳去。更合理的做法是维护一个“上一轮命中的能力”，在用户没有明显切换意图时优先沿用。

这就是子流程粘滞的价值：让多轮交互更像“继续刚才的事”，而不是“每次重新开题”。

## 亮点五：只有成功轮次才写入粘滞状态

粘滞状态不是越早写越好。一个常见误区是：只要路由出来了，就立即把能力状态持久化。

更稳的语义是：

- 取消轮次不写
- 出错轮次不写
- 只有真正成功完成的轮次才更新状态

原因很简单：失败和取消都不该成为后续对话的“历史事实”。否则用户中途打断一次，或者系统半途失败一次，后面就容易被错误带偏。

## 为什么这是通用亮点

这个模式真正有价值的地方在于，它把 Agent 从“全靠模型理解上下文”变成了“模型判断 + 工程控制”的组合：

- 能力可扩展
- 路由可降级
- 多轮体验更稳定
- 失败和取消不会污染状态

这类设计对任何多能力 Agent 都适用，不局限于某个业务域。

## 适用场景

- 多技能对话机器人
- 表单填写 / 配置修改类 Agent
- 同时支持问答、检索、操作的智能助手
- 有长链路、多轮上下文的任务型对话系统

## 使用建议

1. 为能力定义稳定名字和元信息
2. 路由器一定要有默认能力
3. 不要把粘滞状态写得过早
4. 路由层要做输出清洗和容错
5. 把“分类正确率”之外的稳定性也纳入设计目标
