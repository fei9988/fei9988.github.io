# 结构化产物通道设计

## 这个设计解决什么问题

很多 Agent 或流式系统不只输出文本，还会输出结构化结果，比如：

- 表单草稿
- 推荐卡片
- 检索结果列表
- 图表配置
- 可执行参数

一个常见但不太好的做法，是把这些结构化结果直接塞进正文，或者让每个能力自己拼协议事件。这样很容易导致模型、编排层和前端耦在一起。

更好的思路是：给结构化结果单独设计一条输出通道。

## 亮点一：正文流和结构化结果分离

一个通用模式是：

- 文本正文继续按增量流输出
- 结构化结果走单独的产物通道
- 最后在统一出口汇合

这样做的好处非常实际：

- 文本渲染逻辑更纯粹
- 结构化结果可以被前端稳定消费
- 不需要从自然语言里反解析 JSON

这类分层比“让模型一边说话一边嵌结构化块”更适合工程落地。

## 亮点二：产物抽象保持足够轻

一个实用的产物模型通常不需要很重，只要有两部分就够了：

- `eventType`：这是什么产物
- `payload`：产物内容是什么

例如：

```java
public class Artifact {
    private String eventType;
    private Object payload;
}
```

这个抽象的价值在于，它不绑定某一种具体业务。今天可以发推荐卡片，明天可以发筛选条件，后天可以发图表定义，协议层都不用重写。

## 亮点三：用请求级 ID 做显式索引

如果结构化结果是在工具调用、异步任务、子线程里产生的，就不应该依赖 `ThreadLocal` 保存当前产物。

更稳的做法是按请求级 ID 显式挂载：

```java
artifactStore.add(requestId, artifact);
List<Artifact> artifacts = artifactStore.consume(requestId);
```

这种方式的优点是：

- 不怕线程切换
- 支持一轮产生多个结果
- 生命周期清晰，便于统一清理

对于“请求过程中的临时结果”，显式索引通常比隐式上下文更可靠。

## 亮点四：生产产物和发送产物分离

另一个很值得复用的原则是：

- 工具或子能力负责“生成产物”
- 编排层负责“何时输出给前端”

示例伪代码：

```java
String toolResult = validateAndBuildDraft(input);
artifactStore.add(requestId, new Artifact("form_draft", draft));
return "草稿已生成，请展示给用户确认。";
```

然后在统一出口：

```java
return textStream
        .concatWith(Flux.fromIterable(renderArtifacts(requestId)))
        .concatWithValues(doneEvent());
```

这种职责拆分很重要，因为它避免了：

- 工具层直接依赖前端协议
- 编排层理解具体业务细节
- 前端从正文里猜测结构化结果

## 亮点五：尾部统一输出，协议顺序稳定

很多时候，把结构化结果放在正文之后统一输出，会比中途打断正文更稳定。

常见顺序可以是：

1. 正文增量流
2. 结构化产物事件
3. `done`

这种顺序的好处是：

- 前端状态机简单
- 用户先看到解释，再看到结构化结果
- `done` 永远是最后的成功信号

如果异常或取消发生，也更容易统一清理临时产物，避免脏数据残留。

## 为什么这是通用亮点

这个设计真正抽象出来的是一句话：让“自然语言输出”和“机器可消费结果”各走各的通道。

它带来的好处是：

- 协议更清晰
- 前后端契约更稳定
- 新增结构化能力更容易
- 模型输出和系统输出的职责边界更明确

## 适用场景

- AI 表单填写
- 智能推荐卡片
- 搜索结果、引用结果、候选项展示
- 图表 / 查询条件 / 参数草稿生成
- 任何“文本解释 + 结构化结果”并存的系统

## 使用建议

1. 结构化结果不要混进正文
2. 用稳定的 `eventType` 作为前后端契约
3. `payload` 尽量直接传强类型对象
4. 请求结束后记得清理临时产物
5. 把“生成产物”和“输出产物”拆成两个阶段
