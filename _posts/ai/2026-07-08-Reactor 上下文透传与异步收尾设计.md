# Reactor 上下文透传与异步收尾设计

## 这个设计解决什么问题

响应式系统最大的错觉之一是：代码看起来像一条链，运行时却早就跨了很多线程。只要系统里还依赖请求身份、链路日志、trace 上下文、取消回滚这些能力，就很容易遇到两个问题：

- 线程一切换，请求级上下文就丢了
- 收尾逻辑里混入阻塞操作，把事件循环线程拖慢

所以一个成熟的响应式设计，不能只关心“主链路能跑通”，还要处理好上下文透传和异步收尾。

## 亮点一：先在入口线程显式捕获上下文

如果某些信息原本存在 `ThreadLocal`、MDC 或网关注入的请求上下文里，最稳的方式不是赌它们能自动传下去，而是在进入响应式链路前先显式读出来。

例如：

```java
String requestId = RequestContext.getRequestId();
String userToken = AuthContext.getToken();
String logId = MDC.get("logId");
```

然后再把它们放入响应式上下文，而不是直接在后面的异步线程里现取。

这一步的价值在于：你明确了“哪些是请求级事实”，后续线程怎么切都不再依赖偶然性。

## 亮点二：用 Reactor Context 承载请求级状态

在响应式编程里，请求级上下文更适合放进 Reactor Context，而不是继续散落在 `ThreadLocal` 里。

一个典型写法：

```java
return businessFlow()
        .contextWrite(ctx -> ctx
                .put("requestId", requestId)
                .put("userToken", userToken)
                .put("logId", logId));
```

这样做的好处是：

- 数据跟随响应式链路传播
- 不依赖具体线程
- 更容易和 tracing、logging、鉴权集成

如果某些旧组件仍然只能从 `ThreadLocal` 取值，再通过 `ThreadLocalAccessor` 之类的机制桥接回去，会比全链路硬依赖 `ThreadLocal` 稳定得多。

## 亮点三：阻塞逻辑主动 offload

响应式链路里最危险的不是 `map`、`flatMap`，而是那些“看起来像收尾，实际上会阻塞”的操作，比如：

- Redis / DB 持久化
- 外部 SDK 上报
- 文件写入
- trace finish

这些逻辑如果跑在事件循环线程上，很容易把系统吞吐拉垮。更合理的方式是主动切到合适的线程池。

示例：

```java
Mono.fromRunnable(this::cleanup)
    .subscribeOn(Schedulers.boundedElastic())
    .subscribe();
```

这里的关键认知是：收尾代码也属于生产流量路径，不能因为它写在 `doFinally` 里，就默认它是“轻量”的。

## 亮点四：取消是正式状态，不是附带情况

很多系统对取消的处理很弱，只是连接断了就算完。但对流式系统来说，取消其实是一个完整状态：

- 需要尽快感知
- 需要停止后续工作
- 可能需要回滚本轮临时状态
- 需要正确落到 trace 和日志里

一个比较好的顺序通常是：

1. 标记已取消
2. 回滚本轮临时状态
3. 再做 finish / 上报

这样做的好处是，最终的 trace、审计状态、缓存状态会保持一致，不会出现“用户已经取消，但系统看起来像成功完成”的错位。

## 亮点五：失败和取消都不应污染长期状态

只要系统里存在“长期状态”，例如：

- 会话记忆
- 路由粘滞状态
- 工作流进度
- 最近一次成功结果

那就应该明确：取消和失败是否应写入这些状态。多数情况下，答案都是否。

这是一个很有价值的设计原则：只有真正成功完成的一轮，才成为后续系统可依赖的历史事实。

## 为什么这是通用亮点

这套模式的价值不在 Reactor 本身，而在它抽象出了一类通用问题的处理方式：

- 请求级上下文如何跨线程不丢
- 阻塞收尾如何不影响主链路
- 取消如何进入正式状态机
- 日志、trace、业务状态如何保持一致

这些问题几乎存在于所有响应式在线服务里。

## 适用场景

- Reactor / RxJava 风格的 API 服务
- 流式对话系统
- 带有工具调用和外部依赖的 Agent
- 需要严格处理取消与回滚的长链路任务

## 使用建议

1. 入口线程先提取上下文，再写入 Reactor Context
2. 阻塞操作明确切到专用线程池
3. 把取消当成状态机的一部分设计
4. 不要让失败或取消污染长期状态
5. 把日志、trace、业务状态的收尾顺序想清楚
